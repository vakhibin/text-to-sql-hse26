# Результаты эксперимента

## Эксперимент: Baseline на Spider Dev Set

### Конфигурация эксперимента

| Параметр | Значение |
|----------|----------|
| Датасет | Spider v1.0 |
| Сплит | dev |
| Количество примеров | 1034 |
| Schema linking | Включен |
| Режим генерации | DIRECT (без chain-of-thought) |

### Используемые модели

| Компонент | Модель | Провайдер |
|-----------|--------|-----------|
| Генерация SQL | `openai/gpt-oss-20b` | OpenRouter |
| Эмбеддинги | `qwen/qwen3-embedding-8b` | OpenRouter |

### Параметры моделей

**LLM (gpt-oss-20b):**
- Temperature: 0.0
- Max tokens: 512
- Timeout: 60 секунд

**Schema Linking:**
- Top-k таблиц: 5
- Top-k колонок: 10
- Метрика схожести: Косинусное сходство

### Результаты

| Метрика | Значение |
|---------|----------|
| **Execution Accuracy (EX)** | **64.1%** (663/1034) |
| Exact Match (EM) | 15.4% (159/1034) |
| Valid SQL Rate | 94.1% (973/1034) |

### Время выполнения

| Метрика | Значение |
|---------|----------|
| Общее время | 2ч 50м (10 193 сек) |
| Среднее время на пример | 9.86 сек |
| Schema linking (среднее) | ~5-6 сек |
| Генерация SQL (среднее) | ~1-3 сек |
| Выполнение запроса (среднее) | ~0.002 сек |

### Анализ ошибок

**Невалидный SQL (5.9% = 61 пример):**
- Пустые ответы (только `;`)
- Синтаксические ошибки (пропущенные значения, неполные запросы)
- Галлюцинации названий таблиц/колонок

**Несовпадение результатов выполнения (30% = ~310 примеров):**
- Неправильный тип JOIN (LEFT вместо INNER)
- Лишние колонки в SELECT
- Отсутствующие/неправильные условия WHERE
- Некорректная логика агрегации


### Сравнение: Выбор модели

Ранние эксперименты на 10-30 примерах:

| Модель | EX | Valid SQL | Проблемы |
|--------|-----|-----------|----------|
| `qwen/qwen-2.5-coder-32b-instruct` | 70% | 70% | Пропуск числовых значений (`Age > ;`) |
| `openai/gpt-oss-20b` | 86.7% | 100% | Нет значительных |

Модель `gpt-oss-20b` была выбрана для полной оценки благодаря лучшей обработке извлечения значений из вопросов.

### Оценка стоимости

**Цены gpt-oss-20b:**
- Input: $0.02 / 1M токенов
- Output: $0.10 / 1M токенов

**Примерное использование (1034 примера):**
- Input: ~500 токенов/пример × 1034 = ~517K токенов → ~$0.01
- Output: ~50 токенов/пример × 1034 = ~52K токенов → ~$0.005
- **Итого: ~$0.015** (+ стоимость эмбеддингов)

### Воспроизводимость

Для воспроизведения эксперимента:

```bash
# 1. Настройка окружения
cp .env.example .env
# Отредактируйте .env, добавив OPENROUTER_API_KEY

# 2. Установка зависимостей
uv sync

# 3. Загрузка датасета Spider
uv run python -m app.data.spider_v1

# 4. Запуск оценки
uv run python -m app.evaluation.runner --split dev
```

### Выводы

1. **64.1% EX — приемлемый baseline** для простой системы на основе промптов без few-shot примеров и коррекции ошибок.

2. **Schema linking эффективен** — уменьшает размер промпта, сохраняя релевантный контекст.

3. **Выбор модели важен** — `gpt-oss-20b` лучше обрабатывает извлечение значений, чем `qwen-coder`.

4. **Есть потенциал для улучшения:**
   - Few-shot примеры (+5-10% ожидаемо)
   - Коррекция ошибок при невалидном SQL (+2-3% ожидаемо)
   - Улучшенный schema linking с keyword matching (+3-5% ожидаемо)
